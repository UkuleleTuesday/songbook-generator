# Spike – GCP Cost Savings (Billing 2026-03-01 to 2026-06-30)

**Source data:** Billing account export `My Billing Account Reports
20260301 – 20260630.csv` (filtered total **€13.24**).

## Problem Statement

Identify cost savings opportunities to reduce GCP spend for the Songbook
Generator project. This is a time-boxed analysis of the billing export for the
period **2026-03-01 → 2026-06-30**, cross-referenced against the actual
infrastructure defined in the repository (`deploy-gcs.sh`, `.github/workflows/`,
`.env`).

## TL;DR

Absolute spend is **tiny** — €13.24 across the ~4-month export
(≈ **€3.30/month**, ~€40/yr run-rate). This is **not** an urgent
cost problem. The actionable signal is the **growth rate**: Cloud Storage
**+151%**, Artifact Registry **+51%**, Cloud Scheduler **+63%**
period-over-period. Costs are small today but compounding because of missing
retention / cleanup hygiene. The largest line item (Cloud Run Functions,
€9.30) is currently **fully masked by the free tier** — a latent risk rather
than a present cost.

Every recommended fix is low-effort (config changes / one-liners). The goal is
to **flatten the growth curves and protect free-tier headroom**, not to chase
euros.

### 1. Artifact Registry — €7.98, +51% (highest-ROI fix)

`.github/workflows/deploy.yaml` builds **5 Gen2 Cloud Functions** (api, worker,
cache-updater, tagupdater, drivewatcher) on every push to `main` **and on every
PR** (`opened` / `reopened` / `synchronize`). Each Gen2 build pushes a new
container image into the `gcf-artifacts` Artifact Registry repo.

The `cleanup-pr-resources` job (`deploy.yaml:383`) deletes PR *functions*,
*scheduler jobs*, and *Pub/Sub topics* on PR close — but it **never deletes the
container images** those builds produced. There is **no Artifact Registry
cleanup policy anywhere in the repo** (confirmed by grep). Images therefore
accumulate indefinitely, which explains the steady +51%.

**Fix (one-time, ~€8 → ~€1):** add a cleanup policy to the `gcf-artifacts`
repo (and any `gcf-sources` / `cloud-run-source-deploy` repos):

```bash
gcloud artifacts repositories set-cleanup-policies gcf-artifacts \
  --location=europe-west1 \
  --policy=cleanup-policy.json   # keep N most-recent versions; delete untagged / >30d
```

### 2. Cloud Storage — €4.21, +151% (fastest grower)

Four buckets are created in `deploy-gcs.sh` (step 4):

| Bucket (`.env`) | Lifecycle | Issue |
|---|---|---|
| `…-cdn-europe-west1` | 90-day delete | **Way too long.** Holds *ephemeral* job outputs (`{job_id}/songbook.pdf`); API `JOB_TTL_DAYS=3` (`deploy.yaml:118`). Dead after 3 days, kept 90. |
| `…-cache-europe-west1` | 90-day delete | Worker cache; 90 days is generous for a regenerable cache. |
| `ukulele-tuesday-songbooks` | **none** | Published PDFs/manifests accumulate forever. The "complete" edition regenerates **hourly** (`complete-songbook-cronjob.yaml`) and publishes a fresh dated PDF + manifest on `main` (`generate-songbook.yaml:277`). Primary +151% driver. |
| `ukulele-tuesday-songbooks-logs` | **none** | Access logs (`deploy-gcs.sh:76`) accumulate forever. |

**Fixes:**

- Reduce CDN / cache lifecycle **90d → 7d** (`deploy-gcs.sh:89`). Job outputs
  are ephemeral; the cache is regenerable.
- Add a lifecycle rule to `…-songbooks-logs` (delete > 60d).
- For `…-songbooks`: only publish when content actually changed (dedupe against
  `latest.json`), and/or transition older dated editions to Nearline/Coldline
  while keeping `latest.json` hot.

### 3. Cloud Run Functions — €9.30 gross, €0.00 net (protect this headroom)

The largest gross line, fully offset by the free tier today (`Other savings`
−9.30). The dominant invocation source is `drivewatcher`, scheduled **every
minute** (`deploy.yaml:350`, `* * * * *`) ≈ 43,200 invocations/month — and
**each open PR deploys its own per-minute drivewatcher** as well. Google Drive
polling does not need 60-second latency for a songbook tool.

**Fix:** drop drivewatcher to every 5–15 min → 5–15× fewer invocations and a
comfortable margin under the 2M-invocation free tier as PR/usage volume grows.
Also consider skipping the hourly "complete" regeneration when nothing changed.

### 4. Cloud Scheduler — €1.03, +63%

Cloud Scheduler bills 3 free jobs/month, then $0.10/job. Standing jobs:
`trigger-cache-updater-job` (`deploy-gcs.sh:163`) and `drivewatcher-scheduler`
(`deploy.yaml:347`) — plus **one `drivewatcher-scheduler-pr-N` per open PR**.
Every concurrently-open PR pushes past the free 3.

**Fix:** skip the drivewatcher scheduler job in PR preview deploys (a PR rarely
needs a live drive watcher) → back inside the free tier.

### 5. Noise — Cloud Build & Vertex AI (€0.01 each)

New this period but negligible. No action.

## Options / Recommended Actions

Prioritised by return on effort:

| # | Action | Where | Effort | Impact |
|---|---|---|---|---|
| 1 | Artifact Registry cleanup policy | `gcf-artifacts` repo (+ deploy step) | 15 min | €8 → ~€1; kills +51% growth |
| 2 | CDN/cache lifecycle 90d→7d; add logs-bucket lifecycle | `deploy-gcs.sh` | 15 min | Kills +151% Storage growth |
| 3 | drivewatcher cron 1min → 10min | `deploy.yaml:350` | 1 line | Protects €9.30 free-tier headroom |
| 4 | Skip per-PR drivewatcher scheduler | `deploy.yaml:327` | small edit | Scheduler back under free tier |
| 5 | Dedupe hourly "complete" publish | `generate-songbook.yaml` | medium | Caps songbooks-bucket growth |

**Bottom line:** items 1–2 are the whole game — two config changes that flatten
the only two curves actually growing. Items 3–4 are cheap insurance to keep
Cloud Run Functions at €0 as PR/usage volume rises.

> Note: the Artifact Registry cleanup policy (item 1) and any one-time lifecycle
> updates to already-existing buckets require a `gcloud` run by someone with
> project access; the repo changes only affect newly-provisioned resources.
